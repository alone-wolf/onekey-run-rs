# 跨平台进程控制策略

## 1. 目的

本文件用于评估并约束 `onekey-run-rs` 在 Unix-like 与 Windows 上实现多进程编排的难点、边界和阶段性目标。

用户目标是全平台支持，但这不代表两端必须在首版一次性实现所有高级能力。当前首版范围仅包含 `check`、`up`、`down`，并且只基于 PID / 子进程句柄判断存活状态，不做健康检查。

## 2. 难度评估

结论：在当前收敛后的范围内，全平台支持的难度是“中等偏高，但可控”。

难点不在配置解析或 CLI，而在进程终止语义：

- Unix-like 可依赖信号和进程组，模型相对直接
- Windows 没有与 Unix 信号完全等价的机制
- Windows 下若子进程继续派生后代进程，停止整个进程树需要单独设计
- `down` 跨进程调用意味着必须持久化运行时状态，且要避免误杀已复用 pid 的无关进程

如果继续收敛范围，不做以下能力，则可明显降低复杂度：

- 不做业务 ready / health
- 不做复杂重启策略
- 不做后台 daemon 模式
- 不做单服务热操作

## 3. 平台差异面

### Unix-like

- 可使用 `SIGTERM` / `SIGKILL`
- 可使用进程组实现整组停止
- 对子进程树清理更容易形成稳定语义

### Windows

- 不能照搬 Unix 信号模型
- 需要明确使用何种停止机制
- 若只保存主 pid，可能无法保证把整棵子进程树都停止干净
- 进程句柄、创建时间和树状关系可能需要进入状态文件

## 4. 首版建议能力分层

### L1 必须等价

以下能力建议在 Unix-like 和 Windows 首版都支持：

- `check`
- `up`
- `down`
- 配置解析与依赖图校验
- 子进程启动
- 基于 pid / 句柄的存活检查
- 退出时清理自身直接启动的服务

### L2 可接受的阶段性差异

如果实现复杂度过高，首版可接受以下差异，但必须文档化：

- Unix-like 优先支持整组进程停止
- Windows 首版先保证停止直接启动的主进程，再逐步完善对子进程树的处理
- 平台特定停止信号字段的可选值可能不同

## 5. 推荐实现策略

建议采用“统一抽象 + 平台后端”的方式：

- `process::backend::unix`
- `process::backend::windows`
- 对上暴露统一接口，如 `spawn_process`、`terminate_process`、`is_process_alive`

统一抽象层至少要覆盖：

- 进程启动结果
- 进程标识信息
- 存活检查
- 优雅停止
- 强制停止

## 6. 是否建议直接做全平台

建议结论：

- 可以按全平台目标设计接口和状态文件
- 实现上优先保证 Unix-like 稳定，再补齐 Windows 细节
- 如果 Windows 下“整棵子进程树停止”在首版验证成本过高，可以先将该差异显式列为已知限制，而不是假装等价

这比一开始就把产品定位改成 Unix-only 更稳妥，因为当前功能范围已经明显收敛，复杂度主要集中在进程终止部分，而不是整个项目都跨平台困难。

## 7. 首版必须补充的验证

- macOS / Linux / Windows 各自的基本启动与停止集成测试
- 进程在运行期间自行退出时的状态变化测试
- 重复执行 `down` 的幂等性测试
- pid 失效或状态文件损坏时的错误处理测试

## 8. 待确认问题

- Windows 首版是否必须保证完整子进程树清理
- `stop_signal` 字段是否需要做平台限定
- 是否需要把进程创建时间写入状态文件以规避 pid 复用误判
