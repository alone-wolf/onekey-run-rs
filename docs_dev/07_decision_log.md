# 设计决策日志

本文件用于记录规划设计阶段的关键决策、未决问题和结论状态。它不是会议纪要，而是设计事实来源。

## 状态定义

- `proposed`
- `accepted`
- `rejected`
- `deferred`

## 当前决策项

### D-001 平台支持范围

- 状态：`accepted`
- 决策：目标平台包含 Unix-like 和 Windows，但需先单独评估是否一次性交付完全等价能力。
- 原因：工具面向通用本地开发场景，不适合在产品定位上先排除 Windows 用户。
- 影响：需要补充跨平台进程控制文档，并明确阶段性交付边界。

### D-002 默认失败策略

- 状态：`accepted`
- 决策：首版默认采用 `fail-fast`。
- 原因：当前不做业务健康检查和复杂重启策略，优先保证编排环境的一致性和可预测性。
- 影响：任一关键服务启动失败或运行中异常退出时，整体进入停止流程。

### D-003 配置版本字段

- 状态：`accepted`
- 决策：当前阶段不引入 `version` 字段，待首个稳定 schema 成形后再考虑。
- 原因：当前配置规范仍在迭代，过早引入版本号会制造额外迁移噪音。
- 影响：配置解析逻辑简化，但未来补版本时需要一次迁移设计。

### D-004 首版命令集

- 状态：`accepted`
- 决策：首版命令集为 `init`、`check`、`up`、`down`。
- 原因：`init` 降低了首次使用门槛，`down` 则补齐进程编排生命周期闭环。
- 影响：CLI 契约需要覆盖模板生成和拒绝覆盖行为，运行时仍必须补 runtime state / pid 文件设计。

### D-005 ready 默认语义

- 状态：`accepted`
- 决策：首版不实现健康检查和业务级 ready；仅通过 PID / 子进程句柄判断存活状态。
- 原因：先收敛最小编排闭环，降低运行时复杂度和跨平台差异面。
- 影响：服务“已运行”不等于“业务可用”，该限制需要在文档中明确。

### D-006 `down` 的状态来源

- 状态：`accepted`
- 决策：`down` 依赖项目目录下 `.onekey-run/state.json` 与 `.onekey-run/lock`；状态文件记录项目根目录、实例标识、服务 pid 及平台元数据。
- 原因：项目级运行目录最容易通过当前目录稳定定位，也便于避免跨项目误操作。
- 影响：`up` 必须负责锁管理和增量写状态；`down` 必须校验状态文件和进程实例归属。

## 使用规则

- 新的设计争议必须登记后再进入实现
- 决策状态变更时要补充原因
- 已接受决策应同步更新相关文档，而不是只修改本文件
