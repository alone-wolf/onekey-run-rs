# 运行时状态持久化设计

## 1. 目的

`down` 不是对当前内存中的进程表操作，而是对“此前某次 `up` 启动出来的项目实例”执行停止。因此必须定义稳定的运行时状态落盘协议。

没有这份文档，`down` 的实现会立刻遇到三个问题：

- 去哪里找需要停止的进程
- 如何确认这些进程属于当前项目
- 如何避免误杀 pid 已复用的无关进程

## 2. 状态文件是否必要

结论：必要。

只靠“重新解析配置然后猜测进程”不可行，因为：

- 配置可能已被修改
- 同名服务可能已被其他方式启动
- 单靠服务名无法定位具体进程实例

## 3. 建议落盘内容

建议为每次 `up` 维护一个项目级状态文件，至少包含：

- `project_root`
- `config_path`
- `started_at`
- `services`
- `tool_pid`
- `instance_id`

每个服务项至少包含：

- `service_name`
- `pid`
- `started_at`
- `cwd`
- `executable`
- `args`
- `log_file`
- `platform_metadata`

其中 `platform_metadata` 用于记录平台特定字段，例如：

- Unix-like 的进程组 id
- Windows 的进程创建时间或其他识别信息

## 4. 建议文件布局

需要先确定状态文件是放在项目目录内，还是放到用户态运行目录。

建议优先考虑：

- 项目目录下的 `.onekey-run/`

建议至少包含：

- `.onekey-run/state.json`
- `.onekey-run/lock`

这样有几个好处：

- `up` 与 `down` 都能通过当前目录稳定定位
- 项目隔离清晰
- 不需要额外全局索引

## 5. 锁与并发语义

必须明确：

- 同一项目目录是否允许多个 `up` 并发执行
- `up` 执行时，另一个 `down` 是否允许抢占

首版建议：

- 同一项目同一时刻只允许一个活动实例
- 通过锁文件或文件锁防止重复 `up`
- `down` 需校验锁和状态文件的一致性

## 6. `up` 的状态写入时机

建议流程：

1. 创建项目运行目录和锁
2. 写入初始状态文件
3. 每启动一个服务，就增量更新状态
4. 进入主循环后继续维护最终状态

这样即使中途失败，也能让清理逻辑拿到尽可能完整的状态。

## 7. `down` 的停止流程

建议流程：

1. 读取状态文件
2. 校验 `project_root` 与当前目录匹配
3. 校验每个服务记录是否仍指向原始进程实例
4. 按依赖逆序停止
5. 删除状态文件和锁

第 3 步非常关键。首版至少要考虑 pid 复用问题，而不是看到 pid 存在就直接杀。

## 8. 幂等性要求

`down` 应尽量设计成幂等：

- 没有状态文件时给出可理解提示，不应视为异常崩溃
- 某个服务已退出时，不应阻塞整个停止流程
- 多次执行 `down` 应有稳定行为

## 9. 与跨平台实现的关系

本文件和 [08_cross_platform_strategy.md](/Users/wolf/RustroverProjects/onekey-run-rs/docs_dev/08_cross_platform_strategy.md) 强相关。

原因是：

- Unix-like 与 Windows 对“同一个进程实例”的识别方式不同
- Windows 可能需要比 pid 更多的元数据
- 停止整棵子进程树的策略也会影响状态文件内容

## 10. 待确认问题

- 状态文件格式用 JSON、YAML 还是其他格式
- 是否记录 stdout / stderr 日志文件路径
- 是否在 `up` 前台退出后保留状态文件用于 `down`
- 是否允许用户手动指定运行时目录
