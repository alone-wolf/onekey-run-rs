# 工程约定

本文件列出在规划设计阶段就必须落盘的工程约定。没有这些约定，后续实现阶段很容易出现风格漂移和行为分裂。

## 1. 命名约定

- 二进制名统一使用 `onekey-run`
- 配置文件名统一使用 `onekey-tasks.yaml`
- Rust 模块命名使用 `snake_case`
- 服务名建议限制为小写字母、数字和连字符

## 2. 平台约定

- 平台目标为同时支持 Unix-like 和 Windows，但允许采用分阶段交付
- 若某能力仅在部分平台可用，需在文档和错误信息中显式说明
- 平台差异不允许隐藏在实现细节中

## 3. 错误处理约定

- 用户可见错误必须可读，不直接暴露底层库堆栈
- 内部错误类型应可区分配置错误、运行时错误和系统错误
- CLI 退出码必须可预测，并与文档保持一致
- 错误上下文要带服务名、字段名或文件路径

## 4. 日志约定

- 内部日志使用统一日志框架
- 服务日志必须带服务名前缀
- 用户输出与调试输出分层
- 默认输出以人类可读为主

## 5. 配置约定

- 所有默认值必须集中定义并文档化
- 未知字段的处理策略必须统一
- 当前阶段不引入 `version` 字段；schema 稳定后再补版本管理
- 新字段上线前必须补文档和示例

## 6. 测试约定

- 配置解析和依赖图必须有单元测试
- 启停、存活检测、失败回滚必须有集成测试
- 影响 CLI 契约的变更必须补充命令级测试
- 修复 bug 时应补最小回归测试

## 7. 依赖管理约定

- 优先选用稳定、维护活跃的 Rust 库
- 引入新依赖时需说明用途和替代方案是否被评估
- 不为小问题引入重量级框架

## 8. 文档约定

- 对外契约变更必须同步更新 `docs_dev`
- 未决问题进入决策日志，不依赖口头同步
- 示例配置必须可运行，不能只是占位
- 涉及跨平台语义的变更必须同步更新平台差异文档

## 9. 发布约定

- 在对外发布前，CLI 命令名、配置字段名和退出码要冻结
- 首版发布说明必须列出已支持平台和未支持能力
- 配置兼容性策略要写入用户文档

## 10. 代码审查关注点

评审时优先关注：

- 是否破坏既有 CLI / 配置契约
- 是否引入不一致的生命周期语义
- 是否补了对应测试
- 是否把默认行为藏在代码里而不是文档里
